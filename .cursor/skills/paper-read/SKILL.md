# 论文精读技能

## 能力概览与整体框架

本技能对单篇学术论文进行**逐句级**深度分析，产出结构化分析报告，使读者无需再读原文即可掌握方法核心、所有图表含义与实验结论。

| 维度 | 说明 |
|------|------|
| **输入** | 1️⃣ 优先读取本地翻译稿：`papers/{论文简称}/{论文简称}_translation.md`。2️⃣ 若不存在，则支持用户提供 **arXiv/论文 URL** 或 **本地 PDF 路径**。3️⃣ 若用户仅给出论文名称或简称，**必须先执行 Step -1（技术报告查找）** 在 arXiv、GitHub、厂商官网等渠道搜索并获取原文/技术报告。 |
| **输出** | 同一目录下 `{论文简称}_analysis.md`，包含：逐句翻译（中英对照，覆盖全文每一句）、方法核心、图与表详解、详细总结。 |
| **逐句翻译** | 第一章之前：除参考文献外，论文**每一句**均提供中英文对照翻译。**严禁**仅翻译要点/摘要/关键句——必须逐句覆盖，一句不漏。 |
| **章节一：方法核心** | 方法动机、方法设计（pipeline/模块/公式）、与 SOTA 对比、实验表现与优势、学习与应用、一句话总结 + 自明性速记 pipeline。每个子节的解释必须**详尽透彻**，讲清原理、为什么这样做、具体怎么做。 |
| **章节二：图与表** | 对论文中**每一个** Figure 和 Table 给出对应小节：图类型、图中元素含义、与正文对应关系；表格列名、指标、结论与论证作用。**架构图**需单独满足「架构图专门规范」（见下），含亮点、改动、改动原因与作用、如何达成效果、达成了什么效果。 |
| **章节三：详细总结** | 基本信息、技术背景与挑战、论文亮点与贡献、方法详解（训练流程/主要架构图/主要方法等用 1、2、3 步骤逐步说明）、实验设置与结果分析、结论（分点 + 一句话）。 |
| **可选能力** | 可调用 **scholar** 子代理搜索相关前沿资料，补充技术背景与相关工作；分析主体仍严格基于论文内容，技术解析准确、避免主观臆断。 |

所有产出文件保存在**项目根目录下的 `papers/{论文简称}/`** 目录中，便于管理。

## 工作流程

### Step -1：技术报告查找（每篇论文必须执行）

在开始精读之前，**必须**先尝试获取该论文的完整技术报告/原文：

1. **arXiv 搜索**：用论文标题、模型名称在 arXiv 搜索（`arxiv.org/search`、`arxiv.org/abs/{id}`），获取 PDF 或 HTML 全文链接。
2. **GitHub 搜索**：在论文对应的 GitHub 仓库中查找 `docs/`、`technical_report.pdf`、`paper.pdf` 等文件；许多模型的技术报告以 PDF 形式放在仓库 `docs/` 或 `main/` 分支下。
3. **厂商官网/博客**：检查已知厂商页面（如 Qwen 官网、OpenBMB 博客、Nvidia 项目页等），获取技术报告或详细博客文档。
4. **WebSearch 辅助**：若以上直接链接未命中，用 `WebSearch` 工具搜索 `"{模型名}" technical report` 或 `"{模型名}" paper arxiv`。

**产出要求**：
- 在分析报告开头的「基本信息」中列出所有找到的技术报告来源链接（arXiv、GitHub PDF、官网等）。
- 若找到多个来源，以内容最完整的版本为精读依据。
- 若确实无法找到任何技术报告，在报告中明确说明「未找到独立技术报告，以下基于可获取的公开信息分析」。

---

### Step 0：逐句翻译（对应报告第一章之前）

在「方法核心」章节之前，提供**逐句翻译**：

- 除**参考文献列表**外，论文正文的**每一句**均给出中英文对照：先英文原文，再中文译文。
- 按小节（Abstract、Introduction、Method、Experiments、Conclusion 等）分块呈现，**必须覆盖全文每一句**（参考文献列表除外）。
- **⚠️ 关键要求：逐句 ≠ 要点**。不能只翻译"关键句"或"要点"，必须像原文一样一句一句翻译，包括过渡句、背景描述、实验细节描述等所有句子。
- 每一句的翻译应忠实原文，术语准确，语句通顺。
- 对于长论文（>20 页），可分批次完成，但最终报告必须包含完整逐句翻译。

---

### Step 1：理解论文的核心（对应报告第一章）

精读论文 **Method** 部分，自我验证理解充分后再总结。必须覆盖以下子点。**每个子节的解释必须详尽透彻**——不能只列要点，要讲清楚"是什么、为什么、怎么做、效果如何"。

#### 1. 方法动机

- 作者为什么提出该方法？驱动力是什么？**详细解释**背景和动因，不能一句带过。
- 现有方法的痛点/不足是什么？**逐条列出**具体局限性，并解释每条局限为什么是问题。
- 研究假设或直觉是什么？用简洁语言概括，但要**解释假设背后的推理逻辑**。

#### 2. 方法设计

- 给出清晰的 **方法流程总结（pipeline）**：输入 → 处理步骤 → 输出，逐步解释每一步的具体操作和技术细节；此部分须**极其细致**，是用户主要阅读目标。
- 若涉及模型结构：描述每个模块的功能与作用，以及它们如何协同工作。**每个模块至少 3-5 句话详细解释**，不能只有一句概括。
- 若有公式/算法：用通俗语言解释其含义以及在方法中的角色。**公式中每个符号的含义都要解释**。
- 每一步操作要解释清楚：为什么这样设计？这样做的好处是什么？与其他可能方案相比有何优势？

#### 3. 与其他方法对比

- 本方法与现有主流方法相比，本质不同在哪里？**逐项分析**，不能笼统概括。
- 创新点与贡献度。
- 在什么场景下更适用？用表格总结：方法对比（优点/缺点/改进点），对比项清晰。

#### 4. 实验表现与优势

- 作者如何验证有效性？实验设计与设置。
- 在哪些指标上超越对比方法？列出最具代表性的**关键数据**（具体数值）与结论。
- 哪些场景/数据集下优势最明显？提供**具体数值证据**。
- 局限性（泛化能力、计算开销、对特定数据的依赖等）；指出论文中承认或隐含的不足。

#### 5. 学习与应用

- 论文是否开源？复现该方法的关键步骤是什么？
- 需注意的超参数、数据预处理、训练细节；实现层面建议。
- 能否迁移到其他任务？若能，如何迁移？

#### 6. 总结

- 用**一句话**概括方法核心思想（不超过 20 字）。
- 给出 **1 个速记版 pipeline**（3–5 个关键步骤），使用自明性表述、避免论文专有名词，使读者仅看 pipeline 即可大体理解论文内容；不要用比喻，直白说明。

---

### Step 2：理解所有图例和表格（对应报告第二章）

对论文中**每一个** Figure 和 **每一个** Table 都必须有对应小节，不得遗漏。

#### 2.1 图（Figure）的解析要求

对**每个图**均需说明：

- **类型**：架构图 / 流程图 / 曲线图 / 效果图 / 示意图等。
- **图中元素**：坐标轴、坐标轴标签、图例项、颜色、线型、标记样式、图表标题与子标题、箭头、框、标注等含义。**每个可见元素都要解释**，不能遗漏。
- **与正文对应**：该图对应正文哪一段、哪一节、哪一公式或算法，如何支撑论述。
- **解读**：这张图在说明什么问题？读者应从中得出什么结论？

#### 2.2 架构图专门规范

对属于「架构图」的图（模型结构、系统框图、数据流图等），除满足 2.1 外，**必须**额外包含：

- **整体结构**：从左到右或从上到下的数据流或阶段划分；各阶段/块之间的先后关系。
- **每个模块**：模块名称、功能、输入与输出、与前后模块的衔接方式。**每个模块单独解释至少 2-3 句话**。
- **关键符号**：箭头表示数据流还是梯度流、虚线/实线的含义、颜色区分（若存在）的含义。
- **与 Method 的对应**：图中每个模块对应论文 Method 中哪一小节、哪一公式或哪一算法，便于对照阅读。
- **亮点**：该架构图体现的本方法的主要亮点是什么。
- **改动**：相对基线或已有方法，图中哪些部分是新增或改动的；**改动的原因与作用**（为什么这样改、解决什么问题）。
- **如何达成效果**：这些改动在技术上如何实现、通过什么机制达成效果。
- **达成了什么效果**：在指标、能力或现象上具体带来了什么提升或变化（结合正文与实验）。

#### 2.3 表（Table）的解析要求

对**每个表**均需说明：

- 表格标题、列名、行名的含义。
- 数据含义、统计指标（如均值、标准差、置信区间等）。
- **关键数据解读**：最好/最差的结果分别是什么？差异有多大？说明什么问题？
- 表格在论证中的作用：支撑什么结论、与哪些实验或图表呼应。

#### 2.4 图例分析（综合）

- 流程图：结合正文说明流程各步含义与分支逻辑。
- 架构图：结合正文逐模块解释，并满足「架构图专门规范」。
- 效果图：说明比较对象与正文结论的对应关系。

---

### Step 3：详细总结（对应报告第三章）

按以下结构组织，信息浓缩但前提要介绍清楚。**方法相关部分必须用 1、2、3… 步骤逐步说明**，便于读者按顺序理解。

- **基本信息**：论文简称、作者列表、发布日期、链接（含 arXiv、GitHub 技术报告 PDF、官网等所有来源）。
- **技术背景与挑战**：问题定义、现有方法局限、核心技术挑战；60–80 字浓缩。
- **论文亮点与贡献**：核心创新点、技术亮点、与 SOTA 的差异；80–100 字浓缩。
- **方法详解**（以下各项均用**步骤化**表述，如 1、2、3…）  
  - **训练流程**：按 1→2→3→… 写出完整训练流程（数据输入、前向、损失、反向、更新等），每步一句话说明。  
  - **主要架构图**：按 1、2、3… 说明图中主要组成部分或数据流阶段，每步对应图中一块或一阶段。  
  - **主要方法/关键技术**：按 1、2、3… 说明核心算法、重要公式、技术实现细节，逐步讲清，避免大段堆砌。  
  - **实验设置**：数据集描述、评估指标、对比方法（可用列表或分点）。
- **实验结果分析**：主要结果、与 baseline 对比、消融实验结论、效率分析；每个实验说明的结论。
- **结论**：分点总结核心结论；最后用一句话概括全文。

## Step 4：自检自查与返工（质量门禁，每篇必须执行）

精读报告完成后，**必须**按以下检查清单逐项自检。任何一项不通过，**必须立即返工修正**后再次自检，直到全部通过。

### 4.1 检查清单

| # | 检查项 | 通过标准 | 不通过时的返工动作 |
|---|--------|----------|-------------------|
| C1 | **技术报告来源** | 报告开头列出了 arXiv 链接、GitHub 技术报告 PDF 链接、官网链接中至少一项，且链接可追溯 | 执行 Step -1，补充查找并补充来源链接 |
| C2 | **逐句翻译覆盖度** | 翻译部分覆盖论文正文每一句（除参考文献），不是仅翻译要点/关键句/摘要 | 补充遗漏的句子翻译，确保每一句都有 |
| C3 | **逐句翻译质量** | 翻译忠实原文、术语准确、中文通顺，非机械直译 | 修正翻译质量不达标的句子 |
| C4 | **方法设计详细度** | 方法设计（1.2 节）中每个模块/步骤有 ≥3 句话的详细解释，包含"是什么、为什么、怎么做" | 扩充解释不足的模块/步骤 |
| C5 | **公式解释完整度** | 论文中出现的每个重要公式都已列出并逐符号解释 | 补充遗漏的公式及符号说明 |
| C6 | **图表完整性** | 论文中每一个 Figure 和 Table 在第二章都有对应小节 | 补充遗漏的图/表小节 |
| C7 | **架构图规范** | 每个架构图均满足 2.2 节全部 8 个要求（整体结构、每个模块、关键符号、Method 对应、亮点、改动、达成机制、达成效果） | 逐项补充缺失的架构图分析要素 |
| C8 | **数值证据** | 实验部分引用了具体数值（如准确率 xx%、BLEU xx 等），非仅"优于/可比" | 补充具体数值数据 |
| C9 | **解释深度** | 每个技术点的解释让非该方向的研究者也能理解（需解释背景术语与动机） | 为不够透彻的解释添加背景与详细说明 |
| C10 | **报告自洽性** | 仅读报告即可完整理解论文，不需要再翻原文 | 补充使报告自洽所需的背景与上下文 |

### 4.2 自检流程

1. **逐项检查**：按 C1–C10 顺序逐一审查报告。
2. **记录结果**：在报告末尾附上自检表（简写即可），标记每项 ✅ 或 ❌。
3. **返工**：对所有 ❌ 项立即修正，修正后**重新执行一轮自检**。
4. **最终确认**：全部 ✅ 后，在报告末尾写明「自检通过，全部 10 项达标」。
5. **若三轮自检后仍有 ❌**：在报告末尾如实说明哪些项未达标及原因（如论文本身缺乏实验数据等客观原因），供用户判断。

## 输出文件与路径

- **论文与报告根目录**：项目根目录下的 **`papers`** 目录；每篇论文使用子目录 **`papers/{论文简称}/`**。
- **分析报告路径**：`papers/{论文简称}/{论文简称}_analysis.md`
- **翻译稿路径**：`papers/{论文简称}/{论文简称}_translation.md`（若由本技能生成）。
- 所有与该论文相关的文件（原文、翻译稿、分析报告等）均保存在 **`papers/{论文简称}/`** 下。

## 注意事项与行为规则

1. **翻译与格式**：逐句翻译是**真正的逐句**，覆盖全文每一句（参考文献除外）；不允许只翻译要点或关键句。保持中英对照格式。
2. **技术解析深度**：每个技术点必须讲透彻——解释清楚"是什么、为什么、怎么做、效果如何"，让读者无需查阅原文或其他资料即可理解。
3. **语言与结构**：专业、严谨、逻辑性强，完全采用中文；严格按照三章及其子点分段与编号。
4. **数据来源**：所有分析必须基于用户提供的论文内容或本技能获取的原文/翻译；可选使用 scholar 子代理补充背景，但结论仍以论文为准。
5. **输出目标**：用户可能仅阅读本技能生成的分析报告而不阅读论文正文，因此报告须自洽、完整、可替代原文理解。
6. **技术报告查找**：每篇论文在精读前必须尝试在 arXiv、GitHub、厂商官网查找技术报告原文，确保基于最完整的信息精读。
7. **自检返工**：每篇报告完成后必须执行 Step 4 自检，不达标必须返工，不能跳过。
